# 마음이음 — 기술 현황 및 설계

## 0. Postgres 연동 상태 요약 (RAG 포함)

| 항목 | 상태 | 설명 |
|------|------|------|
| **DB 연결** | ✅ 됨 | Prisma + PostgreSQL. 스키마 적용 시 테이블 생성·연동 가능. |
| **대화 저장** | ✅ 됨 | 채팅 시 `Conversation` 생성, `Message`·`HealthLog` 저장하고 있음. |
| **대화 조회(채팅 시)** | ❌ 안 함 | 채팅할 때 **DB에서 메시지를 읽지 않음**. 클라이언트가 보낸 `messages`만 사용. |
| **페이지 새로고침 시** | ❌ 복구 안 됨 | DB에 저장된 과거 메시지를 불러오는 API/화면 없음 → 새로고침하면 대화 비어 보임. |
| **RAG(벡터 검색)** | ⚠️ 구현됨, DB 설정 필요 | pgvector + `message_embeddings` 테이블 생성 후 동작. `docs/RAG_SETUP.md` 참고. |

**정리**: RAG 코드는 반영됨. DB에 pgvector + message_embeddings 테이블 생성 후 동작(docs/RAG_SETUP.md). Postgres는 “저장만” 쓰던 것에 더해, RAG로 과거 대화 검색 후 맥락 반영이 가능합니다.

## 1. RAG(과거 대화 맥락) 현황

### 현재 상태
- **DB**: PostgreSQL + Prisma 연결됨. `Conversation`, `Message` 모델로 대화 **저장** 중.
- **RAG 미구현**: `app/api/chat/route.ts`에 주석만 있음.
  ```ts
  // RAG: 추후 prisma + pgvector로 유사 과거 대화 검색
  // const memories = await searchMemories(session.user.id, lastUserMessage);
  const memories = "";
  ```
- **현재 맥락 공급 방식**:
  - 클라이언트가 **현재 세션의 전체 메시지**를 `messages`로 매 요청마다 전달.
  - 서버는 이를 `historyText`로 이어붙여 프롬프트에 넣어 Gemini에 전달.
  - 따라서 **같은 대화(Session) 안에서는** “방금까지 한 대화”는 이미 프롬프트에 포함됨.
- **한계**:
  - 다른 날/다른 대화의 과거 내용은 전혀 반영되지 않음.
  - pgvector 등 유사도 검색 없음 → “과거에 비슷한 주제로 뭐라고 했는지” 검색 불가.

### 반복 질문 완화 (적용 완료)
- 시스템 프롬프트에 **[중요: 반복 질문 금지]** 블록 추가: “이미 답한 내용(식사·수면·기분 등)은 다시 묻지 말 것”.
- 텍스트/음성 공통으로 “대화 내역에서 이미 답한 것은 묻지 말고, 아직 안 물어본 주제로만 질문”하도록 지시 추가.
- 이로써 **같은 대화 안에서** 저녁/잠/식사 등을 계속 돌려 물어보는 현상은 완화될 수 있음.

### 추후 RAG 확장 방향
1. **pgvector** (또는 외부 벡터 DB) 도입: `Message.content` 또는 요약을 임베딩해 저장.
2. **검색**: 현재 사용자 `userId` + (선택) `conversationId` 기준으로, 최근/유사 과거 메시지 검색.
3. **프롬프트 삽입**: 검색 결과를 `memories` 문자열로 만들어 기존 `memories` 자리에 넣기.

## 2. 목소리 학습(사용자·가족) 설계

### 요구사항 정리
- **사용자 목소리 등록**: 회원가입(또는 온보딩) 시 사용자 본인 목소리를 학습해 두고, 이후 대화에서 “지금 말한 사람이 사용자 본인인지” 구분.
  - 다른 사람(가족, 방문객)이 말한 것을 사용자로 오인하면 잘못된 상태 판단으로 이어질 수 있음.
- **AI(가족) 목소리**: TTS를 가족 목소리로 쓸 경우, 해당 가족 목소리도 등록해 두어 “지금 말한 건 AI(가족) 목소리”로 인식 가능하게 할 필요 있음.

### 기술적 방향

| 구분 | 내용 |
|------|------|
| **사용자 목소리** | 회원가입/온보딩 시 “목소리 등록” 단계에서 일정 시간(예: 15~30초) 또는 3~5문장 녹음 → 서버에 **화자 임베딩(speaker embedding)** 또는 **Voice ID** 형태로 저장. 이후 음성 입력 시 **화자 검증(speaker verification)** 또는 **화자 식별(speaker identification)** 수행 후, “등록된 사용자 목소리일 때만” 사용자 발화로 처리. |
| **가족 목소리** | (1) TTS용: 가족이 녹음한 샘플로 음성 클론/합성 API 연동. (2) 구분용: 같은 기기에서 AI가 말할 때 나오는 음성도 “이 목소리는 AI(가족)”로 등록해 두면, 다중 화자 환경에서 사용자 vs AI 발화 구분에 활용 가능. |

### DB/스키마 제안
- **User** (또는 별도 테이블):
  - `userVoiceEmbedding` 또는 `userVoiceId`: 사용자 목소리 등록 데이터(URL 또는 바이너리 참조).
  - `userVoiceEnrolledAt`: 등록 시각 (선택).
- **가족 목소리** (한 사용자당 한 명의 “AI 가족” 가정):
  - `familyVoiceId` / `familyVoiceSampleUrl`: 가족 목소리 샘플 또는 TTS용 ID.
  - 가족 목소리도 “화자 ID”로 저장하면, 나중에 “지금 말한 건 사용자 / AI(가족)” 구분 로직에 사용 가능.

### 구현 순서 제안
1. **회원가입 플로우**: 이메일·비밀번호·이름 등 기존 단계 뒤에 “목소리 등록” 단계 추가 (녹음 → 업로드 → 서버에서 embedding/ID 생성·저장).
2. **채팅 음성 입력 파이프라인**: STT 전/후에 “현재 음성 = 등록된 사용자?” 검증. 실패 시 “본인 목소리로 다시 말씀해 주세요” 또는 무시.
3. **가족 목소리**: TTS 연동(클론 API 등)과 “AI 목소리” 등록은 별도 작업으로, 회원가입/설정 화면에서 “가족 목소리 등록” 옵션 추가.

### 외부 서비스 참고
- 화자 인식: AWS Lex (speaker ID), Google Cloud Speaker Recognition, Azure Speaker Recognition, 또는 오픈소스(예: pyannote.audio 등)로 embedding 추출 후 자체 검증.
- 음성 클론 TTS: ElevenLabs, Play.ht, Resemble 등 (가족 목소리 대체용).
